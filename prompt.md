# 오목봇 소개/개발동기 블로그 글 작성 위임 프롬프트

아래 지시를 따라, 컴퓨터교육과 학생의 관점에서 오목봇 프로젝트를 소개하는 블로그 글을 작성해줘.
핵심은 "자랑"이 아니라 "무엇을 만들었고, 어떻게 설계했으며, 교육적으로 왜 의미가 있는지"를 독자가 이해하도록 설명하는 것이다.

## 역할
- 너는 교육용 소프트웨어 프로젝트를 설명하는 기술 블로그 에디터다.
- 대상 독자는 다음 3집단이다.
  1. 같은 학과 학생
  2. 프로그래밍 입문자
  3. 교직/교육공학 관점에서 프로젝트를 보는 독자

## 글의 목표
- 이 프로젝트가 단순 게임이 아니라 "학습 가능한 상호작용 시스템"이라는 점을 전달한다.
- 구현 디테일을 과장 없이 설명한다.
- 교육적 목적(문제해결력, 데이터 기반 사고, 피드백 설계)을 분명하게 드러낸다.

## 반드시 포함할 핵심 내용
1. 이게 무엇인지
- 프로젝트명: 오목봇
- 한 줄 정의: 브라우저에서 바로 실행되는 적응형 오목 AI
- 주요 기능: 즉시 실행, 사용자 패턴 학습, 확률 시각화, 승률 추적, 이어하기/내보내기/불러오기

2. 왜 만들었는지(개발 동기)
- 컴퓨터교육과 학생으로서, "게임 제작"보다 "학습 과정을 설계하는 경험"을 얻기 위해 시작했다는 점
- 학생이 체감하는 피드백(승률 변화, 수 추천 확률)을 통해 학습 동기를 높이고 싶었다는 점
- AI를 신비화하지 않고, 휴리스틱 + 데이터 누적으로 점진적으로 개선되는 구조를 직접 구현해 보고 싶었다는 점

3. 어떻게 설계했는지(핵심 설계)
- 프론트엔드 단일 페이지 구조(index.html, style.css, game.js)
- 보드/수 후보 평가: 규칙 기반 휴리스틱
- 적응 레이어: TensorFlow.js 기반 보정 모델
- 저장 전략: localStorage를 이용한 게임 상태/전적/학습 데이터 유지
- 시각화 설계: 확률 히트맵(봇 우선/사용자 예측/봇 선택), 승률 흐름 차트
- 난이도/행동 전략: 전적 또는 누적 게임 수에 따라 봇의 선택 강도 조정

4. 교육적 목적과 학습효과
- 알고리즘 사고: 패턴 평가, 후보군 축소, 점수 기반 의사결정
- 데이터 리터러시: 로그와 통계를 바탕으로 전략을 수정하는 경험
- 인간-컴퓨터 상호작용(HCI): 사용자가 이해할 수 있는 피드백 UI 설계
- 반성적 학습: "왜 이 수를 두었는가"를 돌아보게 하는 설명 가능한 시각화

5. 한계와 다음 개선 계획
- 브라우저 저장소 기반 학습의 한계(기기/브라우저 종속성)
- 모델 일반화 한계(사용자별 편향)
- 향후 계획: 대국 기록 분석 대시보드, 모델 경량화, 교육용 모드(난이도/설명 옵션)

## 글쓰기 스타일 가이드
- 톤: 진솔하고 설명적인 1인칭 복수 또는 1인칭 단수(학생 개발자 시점)
- 금지: 과장 표현("혁신", "압도적"), 근거 없는 성능 주장
- 권장: "의도 -> 구현 -> 관찰 -> 배운 점" 흐름
- 기술 비전공자도 읽을 수 있게 용어는 짧게 풀어서 설명

## 출력 형식
아래 형식을 정확히 지켜 작성해줘.

1) 제목 후보 3개
- 교육적 가치가 드러나는 제목으로 3개

2) 본문(마크다운)
- 분량: 1200~1800자
- 섹션 구성(소제목 필수)
  - 프로젝트 한눈에 보기
  - 개발 동기: 왜 오목봇을 만들었는가
  - 설계와 구현: 어떻게 학습하는 봇을 만들었는가
  - 교육적 의미: 컴퓨터교육 관점에서의 가치
  - 한계와 다음 단계

3) 3줄 요약
- 블로그 상단에 넣을 요약 3문장

4) 태그 8개
- 예: #컴퓨터교육 #프로젝트기록 형태

## 추가 제약
- 코드 전문은 넣지 말고, 필요한 경우 의사코드 수준으로 1~2줄만 언급
- 실제 사용 기술명은 정확히 표기: TensorFlow.js, localStorage, JavaScript
- "재미"보다 "교육적 설계 의도"가 더 강조되도록 작성
- 마지막 문단은 "이 프로젝트를 통해 학생 개발자로서 무엇을 배웠는지"로 마무리
