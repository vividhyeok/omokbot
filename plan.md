# 오목봇 개선 계획

## 전제

이 프로젝트는 순수 연구용 강화학습 엔진이 아니라, **강화학습의 핵심 감각을 브라우저에서 체험시키는 교육용 오목봇**으로 보는 것이 맞다.

Teachable Machine이 지도학습을 쉽게 보여주기 위해 데이터 수집, 모델 선택, 학습 루프를 크게 단순화한 것처럼, 이 오목봇도 강화학습을 그대로 구현하기보다 다음 타협을 이미 포함하고 있다.

- 완전한 탐색/정책 학습 대신 휴리스틱 오목 평가를 사용한다.
- 장기 보상 학습 대신 게임 종료 후 승패 보상을 간단히 반영한다.
- 대규모 자기대국 대신 사용자의 실제 대국 기록을 학습 신호로 쓴다.
- 서버/GPU 없이 브라우저 로컬 저장소와 TensorFlow.js로 작동한다.

따라서 개선 목표는 **타협을 없애는 것**이 아니라, **타협이 겉으로 튀어나오지 않도록 하나의 일관된 교육 경험으로 포장하는 것**이다.

Teachable Machine도 내부 구현의 단순화, 모델 한계, 데이터 전처리 방식을 사용자의 첫 경험에 노출하지 않는다. 대신 사용자가 `예시를 모은다 -> 학습한다 -> 결과가 바뀐다`는 큰 흐름을 바로 이해하게 만든다. 이 오목봇도 마찬가지로 `대국한다 -> 봇이 경험을 저장한다 -> 다음 판의 선택이 달라진다`는 경험을 중심에 둬야 한다.

## 현재 아쉬운 점

### 1. 학습 경험의 흐름이 아직 선명하지 않음

현재 앱은 봇이 학습하는 것처럼 보이지만, 사용자가 "내가 둔 판이 다음 판에 어떤 영향을 줬다"는 감각을 충분히 받기 어렵다. 승률 바와 히트맵은 있지만, 학습 전후 변화가 하나의 이야기로 이어지지 않는다.

### 2. 휴리스틱과 학습 모델의 역할이 섞여 있음

봇의 착수는 휴리스틱 점수, 사용자 패턴 예측, TensorFlow.js 보정값, 강제 방어 로직이 섞여 결정된다. 이 구조 자체는 교육용으로 합리적이지만, 현재는 어느 요소가 결정을 얼마나 좌우했는지 분리해서 설명하지 않는다.

### 3. 학습 루프가 너무 결과 중심임

게임 종료 후 승/패/무 결과를 바탕으로 모델을 업데이트하지만, 중간 수의 기여도나 보상 분배가 단순하다. 사용자는 "왜 방금 판이 학습에 도움이 되었는지" 체감하기 어렵다.

### 4. 학습 요소들이 하나의 경험으로 정리되지 않음

`gomoku-player-model`, 휴리스틱 평가, TensorFlow.js 보정 레이어가 각각 다른 역할을 하지만 사용자는 그 구분을 알 필요는 없다. 문제는 내부 구조를 설명하지 않는 것이 아니라, 이 요소들이 합쳐져서 "봇이 경험을 바탕으로 다음 선택을 바꾼다"는 단순한 경험으로 정리되지 않는 점이다.

### 5. 교육용 실험 도구가 부족함

학습률, 탐험 정도, 휴리스틱 의존도, 사용자 예측 의존도 같은 값을 조절하며 결과를 비교할 수 없다. 그래서 교육용 프로그램이라기보다 "학습하는 봇이 들어간 게임"에 머문다.

### 6. 코드 구조가 실험 확장에 불리함

현재 `game.js` 하나에 게임 규칙, AI 판단, 학습, 저장소, 렌더링, UI 이벤트가 모두 들어 있다. 기능을 계속 추가하면 어느 부분이 게임 엔진이고 어느 부분이 교육 UI인지 구분하기 어려워진다.

## 개선 방향

## 1단계: 학습 경험을 먼저 선명하게 만들기

목표: 사용자가 내부 타협을 몰라도 "내 대국이 봇의 다음 판단에 영향을 준다"는 흐름을 바로 이해하게 만든다.

### 작업

- 사이드 패널에 `학습 상태` 섹션을 추가한다.
- 기술 용어 대신 사용자가 바로 이해할 문구를 쓴다.
  - "이번 판에서 봇이 본 장면"
  - "봇이 기억한 위험한 자리"
  - "다음 판에 더 신경 쓸 패턴"
  - "방금 선택에 영향을 준 기억"
- `이 앱의 타협` 같은 노출형 섹션은 UI에 넣지 않는다.
- README도 첫 화면에서는 "강화학습 교육용 오목봇"의 경험을 설명하고, 구현 한계는 아래쪽의 개발 노트에서만 짧게 다룬다.
- 게임 종료 모달에 이번 판에서 저장된 학습 신호를 요약한다.
  - 봇이 기억한 사용자 공격 패턴
  - 다음 판에 더 조심할 위치
  - 학습에 사용된 봇 착수 수
  - 학습 완료 여부

### 완료 기준

- 사용자가 내부 구현을 몰라도 "봇이 방금 판에서 무엇을 배웠는지" 이해할 수 있다.
- 앱이 변명처럼 보이지 않고, 교육용 도구로 자연스럽게 보인다.

## 2단계: 봇 결정 과정을 분해해서 보여주기

목표: "AI가 왜 저기에 뒀는지"를 사용자가 납득할 정도로만 설명한다.

### 작업

- 봇 착수 후보 상위 5개를 표시하는 `결정 분석` 패널을 만든다.
- 각 후보에 대해 내부 점수를 그대로 노출하지 않고, 사용자가 이해하기 쉬운 항목으로 묶어 보여준다.
  - 전술적으로 좋은 자리
  - 사용자가 노릴 가능성이 높은 자리
  - 이전 대국에서 기억한 자리
  - 반드시 막아야 하는 자리
  - 봇의 최종 선택 가능성
- 현재 히트맵 모드를 다음처럼 정리한다.
  - `전술`: 오목 규칙과 휴리스틱 기준 유리한 자리
  - `상대 예측`: 사용자가 둘 가능성이 높은 자리
  - `정책`: 봇이 실제로 고를 가능성이 높은 자리
  - `가치`: 모델이 유리하다고 보는 자리
- 봇 로그 문구를 단순 감탄형이 아니라 자연스러운 근거형으로 바꾼다.
  - 예: "이 자리는 당신이 자주 이어 가는 흐름이라 먼저 막았습니다."
  - 예: "다음 수에 위험해질 수 있어 방어했습니다."

### 완료 기준

- 같은 착수라도 전술, 기억, 방어 중 무엇이 컸는지 감각적으로 구분할 수 있다.
- 히트맵이 단순 장식이 아니라 학습 개념 설명 도구가 된다.

## 3단계: 교육용 실험 모드 추가

목표: 사용자가 어려운 내부 파라미터가 아니라, 교육용 조작값을 바꾸며 결과를 비교하게 만든다.

### 작업

- `실험 모드` 토글을 추가한다.
- 실험 모드에서 다음 값을 조절할 수 있게 한다.
  - 모험성: 좋은 수만 둘지, 여러 후보를 시도할지
  - 전술 감각: 오목 규칙 기반 판단을 얼마나 강하게 볼지
  - 기억 의존도: 이전 대국의 사용자 패턴을 얼마나 믿을지
  - 학습 속도: 새 판 결과를 얼마나 빠르게 반영할지
  - 학습 사용 여부: 경험 저장을 켜거나 끄기
- 프리셋을 제공한다.
  - `초보 실험`: 탐험 높음, 휴리스틱 약함
  - `균형`: 현재 기본값에 가까운 설정
  - `전술 우선`: 휴리스틱 강함
  - `상대 분석 우선`: 사용자 예측 강함
- 설정 변경 전후 승률 흐름과 선택 분포를 비교한다.

### 완료 기준

- 사용자가 "모험성과 안정성", "전술과 기억"의 차이를 직접 체감할 수 있다.
- 교육용 프로그램으로서 조작 가능한 실험 변수가 생긴다.

## 4단계: 학습 데이터를 더 정직하게 관리하기

목표: 학습 데이터가 불투명한 마법처럼 보이지 않도록, 필요한 수준의 요약만 제공한다.

### 작업

- 저장 데이터에 버전을 붙인다.
  - 예: `gomoku-schema-version`
- `봇의 기억` 화면을 만든다.
  - 총 대국 수
  - 가장 자주 둔 사용자 위치
  - 자주 반복된 사용자 이동 패턴
  - 위협 위치로 기록된 좌표
  - 최근 loss 변화
- 내보내기 파일에 사람이 읽을 수 있는 메타데이터를 추가한다.
  - 생성 시각
  - 앱 버전
  - 총 대국 수
  - 모델 구조 버전
- 가져오기 전에 호환성 검사를 한다.
- 전체 초기화 외에 다음 초기화 옵션을 분리한다.
  - 현재 판만 초기화
  - 사용자 패턴만 초기화
  - 신경망 가중치만 초기화
  - 전체 초기화

### 완료 기준

- 사용자가 봇이 어떤 경험을 기억하고 있는지 요약해서 확인할 수 있다.
- 저장 데이터가 꼬였을 때 전부 날리지 않고 필요한 부분만 초기화할 수 있다.

## 5단계: 강화학습 느낌을 더 살리는 학습 루프 개선

목표: 현재의 단순 승패 보상 구조를 더 자연스러운 "경험 학습" 흐름으로 만든다.

### 작업

- 내부적으로 경험 저장소를 명시적으로 만든다.
  - `state`: 착수 전 보드
  - `action`: 봇 착수
  - `reward`: 최종 또는 중간 보상
  - `nextState`: 착수 후 보드
  - `done`: 게임 종료 여부
- 보상 설계를 단계적으로 개선한다.
  - 승리: +1
  - 패배: -1
  - 무승부: 0
  - 즉시 위협 차단: 작은 양수 보상
  - 상대 즉시 승리 허용: 작은 음수 보상
  - 열린 4 생성: 작은 양수 보상
- 게임 종료 후 전체 기록을 한 번에 학습하는 방식은 유지하되, UI에서는 `방금 판을 복습하는 중`처럼 자연스러운 표현을 쓴다.
- 가능하면 간단한 TD 형태의 목표값을 추가한다.
  - `target = reward + gamma * nextValue`
  - 브라우저 성능을 고려해 작은 후보 집합에만 적용한다.
- 개발자/고급 실험용으로만 `학습 방식` 옵션을 둔다.
  - 결과 중심 학습
  - 위험/방어 기억 포함
  - 다음 장면 예측 포함

### 완료 기준

- 코드에는 `경험`, `보상`, `다음 상태`, `에피소드` 개념이 드러난다.
- UI에서는 그 개념을 쉬운 언어로 번역해 보여준다.

## 6단계: 코드 구조 정리

목표: 실험 기능을 추가해도 코드가 무너지지 않게 역할을 나눈다.

### 제안 파일 구조

```text
game.js                  # 초기 연결 및 앱 부트스트랩
src/rules.js             # 오목판, 착수, 승패 판정, 후보 수
src/evaluator.js         # 휴리스틱 평가
src/playerModel.js       # 사용자 패턴 모델
src/learningModel.js     # TensorFlow.js 모델 생성/저장/학습
src/policy.js            # 봇 후보 점수 합성 및 선택
src/storage.js           # localStorage import/export/versioning
src/renderBoard.js       # 캔버스 보드 렌더링
src/ui.js                # 패널, 모달, 버튼 상태
src/education.js         # 학습 상태, 결정 분석, 실험 모드
```

정적 HTML 실행을 유지하려면 ES module을 쓰거나, 빌드 없는 여러 `<script type="module">` 구조로 나눈다.

### 완료 기준

- 게임 규칙과 AI 정책을 UI 코드 없이 테스트할 수 있다.
- 새 학습 방식이나 새 히트맵을 추가할 때 `game.js` 전체를 건드리지 않아도 된다.

## 7단계: 검증과 교육 품질 개선

목표: "그럴듯해 보이는 앱"에서 "설명 가능한 교육 도구"로 끌어올린다.

### 작업

- 핵심 로직 테스트를 추가한다.
  - 5목 승리 판정
  - 후보 수 생성
  - 즉시 승리 수 탐지
  - 사용자 패턴 업데이트
  - 보상 계산
- 샘플 시나리오를 만든다.
  - 사용자가 같은 초반 수를 반복할 때 예측 히트맵이 변하는지
  - 봇이 패배한 공격 좌표를 다음 판에서 더 빨리 막는지
  - 탐험 정도를 낮추면 선택 분포가 좁아지는지
- README 하단의 개발 노트에 교육용 한계를 짧게 적는다.
  - "이 앱은 AlphaZero류 구현이 아님"
  - "브라우저 실시간 체험을 위해 탐색과 학습을 제한함"
  - "강화학습의 전체 성능보다 개념 이해를 우선함"

### 완료 기준

- 기능 변경 후 기본 규칙이 깨졌는지 빠르게 확인할 수 있다.
- 프로젝트 소개 문서가 과장 없이 설득력 있게 바뀐다.

## 최종 목표

이 프로젝트의 최종 형태는 **"강화학습을 완벽히 구현한 오목 엔진"**이 아니라, **"대국 경험을 통해 봇의 판단이 바뀌는 과정을 직관적으로 체험하는 교육용 오목 실험실"**이어야 한다.

핵심은 성능보다 체험의 명료함이다. 내부 타협을 전면에 설명하기보다, 봇이 무엇을 기억했고 다음 선택이 어떻게 달라졌는지 사용자가 자연스럽게 느끼게 해야 한다.
